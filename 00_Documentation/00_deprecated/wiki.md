# PMI Project Management Process Groups

## 1. Initiating

**Purpose:** Define the project and obtain authorization to start.

**Key Activities:**

- Define project objectives
- Identify stakeholders
- Develop the Project Charter
- Assess feasibility

**Main Deliverables:**

- Project Charter
- Stakeholder Register

---

## 2. Planning

**Purpose:** Establish the roadmap for achieving the project objectives.

**Key Activities:**

- Define scope and requirements
- Develop schedule and budget
- Plan resources and communications
- Identify risks
- Define quality standards

**Main Deliverables:**

- Project Management Plan
- Scope Statement
- Schedule
- Budget
- Risk Management Plan

---

## 3. Executing

**Purpose:** Perform the work defined in the project plan.

**Key Activities:**

- Manage the project team
- Execute project tasks
- Manage stakeholders
- Ensure quality standards
- Procure resources

**Main Deliverables:**

- Project Deliverables
- Work Performance Data
- Issue Logs

---

## 4. Monitoring and Controlling

**Purpose:** Track performance and ensure the project stays on course.

**Key Activities:**

- Measure performance
- Control scope, schedule, and cost
- Manage risks
- Approve changes
- Report progress

**Main Deliverables:**

- Performance Reports
- Change Requests
- Updated Plans

---

## 5. Closing

**Purpose:** Formally complete the project or phase.

**Key Activities:**

- Obtain final acceptance
- Close contracts
- Document lessons learned
- Release resources

**Main Deliverables:**

- Final Project Report
- Lessons Learned Register
- Project Closure Document

---

## PMI Lifecycle Overview

> **Initiating → Planning → Executing → Monitoring & Controlling → Closing**

A key PMI principle is that **Monitoring & Controlling happens continuously during Execution**, not only after it.

---

# Applying PMI Process Groups to Agile

PMI itself (especially in the _Agile Practice Guide_ and _PMBOK 7th Edition_) emphasizes that the five process groups still apply, but they are **iterative and adaptive** rather than linear.

---

## 1. Initiating in Agile

### How it Applies

Agile projects still require formal initiation, but with **lighter upfront definition**.

**Typical Agile Activities:**

- Define product vision
- Identify stakeholders
- Create a high-level roadmap
- Define success criteria
- Establish team structure
- Initial product backlog

**Key Deliverables:**

- Product Vision Statement
- Initial Product Backlog
- Agile Charter or Lean Business Case

### Key Difference from Traditional

| Traditional                  | Agile                                          |
| ---------------------------- | ---------------------------------------------- |
| Detailed scope defined early | Scope evolves over time                        |
|                              | Only high-level requirements defined initially |

**PMI Concept:** _"Progressive elaboration"_

---

## 2. Planning in Agile

### How it Applies

Planning happens **continuously**, not once.

**Typical Agile Planning Levels:**

| Level              | Example        |
| ------------------ | -------------- |
| Vision Planning    | Product goals  |
| Release Planning   | Major features |
| Iteration Planning | Sprint backlog |
| Daily Planning     | Standups       |

**Key Deliverables:**

- Product Backlog
- Sprint Backlog
- Release Plan
- Definition of Done

### Key Difference from Traditional

| Traditional Planning                     | Agile Planning                     |
| ---------------------------------------- | ---------------------------------- |
| Full schedule and budget defined upfront | Rolling-wave planning              |
|                                          | Plan only near-term work in detail |

---

## 3. Executing in Agile

### How it Applies

Execution occurs in **short iterations (Sprints)**.

**Typical Activities:**

- Develop product increments
- Daily standups
- Collaboration with stakeholders
- Continuous testing
- Continuous integration

**Key Deliverables:**

- Working Product Increment
- Updated Backlog
- Sprint Burndown

### Key Difference from Traditional

| Traditional Execution          | Agile Execution            |
| ------------------------------ | -------------------------- |
| Deliverables often appear late | Frequent delivery of value |

**PMI Emphasis:** _"Value delivery over activity completion"_

---

## 4. Monitoring & Controlling in Agile

### How it Applies

Monitoring is **built into Agile ceremonies**.

**Typical Agile Controls:**

| Method            | Purpose           |
| ----------------- | ----------------- |
| Daily Standups    | Track progress    |
| Sprint Review     | Validate results  |
| Retrospective     | Improve process   |
| Burndown Charts   | Track schedule    |
| Velocity Tracking | Forecast delivery |

**Key Deliverables:**

- Velocity metrics
- Burndown charts
- Retrospective actions

### Key Difference from Traditional

| Traditional           | Agile                                       |
| --------------------- | ------------------------------------------- |
| Formal status reports | Visual and continuous control               |
|                       | Example: Task board instead of long reports |

---

## 5. Closing in Agile

### How it Applies

Closing happens at **two levels**:

**Iteration Closing:**

- Sprint review
- Sprint retrospective
- Acceptance of increment

**Project Closing:**

- Final product release
- Lessons learned
- Documentation
- Team release

### Key Difference from Traditional

| Traditional       | Agile                                                  |
| ----------------- | ------------------------------------------------------ |
| One final closure | Multiple mini-closures                                 |
|                   | Every sprint includes: Review, Acceptance, Improvement |

---

## Key PMI Differences: Agile vs Predictive

| Area                 | Predictive (Waterfall) | Agile           |
| -------------------- | ---------------------- | --------------- |
| Scope                | Fixed                  | Flexible        |
| Planning             | Upfront                | Continuous      |
| Change               | Controlled             | Expected        |
| Delivery             | End of project         | Incremental     |
| Documentation        | Heavy                  | Lightweight     |
| Customer Involvement | Periodic               | Continuous      |
| Success Measure      | Scope completion       | Value delivered |

### Important PMI Insight

PMI stresses that **Agile does NOT replace the five process groups**.

Instead:

> **Agile = Iterative application of Initiating → Planning → Executing → Monitoring → Closing**

Every Sprint includes:

- Mini-planning
- Mini-execution
- Mini-control
- Mini-closing

### The Biggest Conceptual Difference (PMI Perspective)

- **Traditional PM** focuses on: _Delivering the planned scope_
- **Agile** focuses on: _Delivering maximum business value_

# Tool Selection in Agile Project Management

In Agile project management (aligned with PMI and the _Agile Practice Guide_), the selection of tools (e.g., backlog tracking software, requirements tools, collaboration platforms) is typically defined primarily during the **Initiating** and early **Planning** activities, and then refined iteratively as the project evolves.

However, Agile treats tool selection differently than traditional projects — tools are not always fixed upfront and may evolve based on team needs.

---

## Primary Stage: Initiating (PMI Process Group)

### Why Tools Are Defined Here

During Initiating, the team establishes the project environment and working approach, including:

- Agile framework selection (Scrum, Kanban, SAFe, etc.)
- Team structure
- Governance model
- High-level infrastructure
- Initial toolchain

**Typical Decisions Made:**

- Backlog management tool (e.g., Jira, Azure DevOps)
- Requirements tracking approach
- Communication tools
- Documentation repositories
- Version control systems
- CI/CD platforms

### PMI Alignment

This falls under:

- Develop Project Charter
- Identify Stakeholders
- Define Project Approach

These decisions may appear in:

- Agile Charter
- Working Agreements
- Project Management Plan (if hybrid)

---

## Secondary Stage: Planning (Iteration Zero / Setup Phase)

### Why Planning Is Critical for Tool Definition

Most Agile teams finalize tool selection during early Planning, often called:

- **Iteration 0**
- **Sprint 0**
- **Project Setup Phase**
- **Environment Setup**

**Typical Activities:**

- Configure backlog tool
- Define workflows
- Configure issue types
- Establish requirement templates
- Set Definition of Done
- Setup repositories
- Setup test environments

**Examples:**

| Area             | Example Tool Decision            |
| ---------------- | -------------------------------- |
| Backlog Tracking | Jira vs Azure DevOps             |
| Requirements     | Confluence vs DOORS vs Git-based |
| Source Control   | GitHub vs GitLab                 |
| Testing          | TestRail vs integrated testing   |
| Documentation    | Wiki vs SharePoint               |

---

## PMI-Agile Interpretation

From a PMI perspective, tool selection belongs to:

### Initiating — High-Level Decisions

- Tool categories
- Constraints
- Budget approval
- Example: _"We will use a cloud-based backlog system."_

### Planning — Detailed Decisions

- Exact tools
- Configuration
- Access setup
- Example: _"We will use Jira with Scrum boards and story points."_

---

## Important Agile Principle

From the Agile Manifesto:

> _"Individuals and interactions over processes and tools."_

**Meaning:**

- Tools support the team
- Tools are not the primary driver
- Tools can change if needed

---

## Practical PMI-Agile Answer

> **Tools are initially identified during Initiating and defined/configured during Planning.**

---

## Real-World Best Practice

A mature Agile setup usually defines tools during the **Project Setup Phase**.

**Typical sequence:**

1. **Initiating** — Choose Agile approach, define constraints, identify tool categories
2. **Early Planning / Sprint 0** — Select tools, configure systems, train team
3. **Execution** — Adjust tools if needed

---

## Key Difference from Waterfall

| Waterfall                            | Agile                             |
| ------------------------------------ | --------------------------------- |
| Tools fixed early and rarely changed | Tools defined early but adaptable |
